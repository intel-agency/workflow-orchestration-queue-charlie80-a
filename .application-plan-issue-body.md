# workflow-orchestration-queue – Complete Implementation

## Overview

workflow-orchestration-queue is a groundbreaking **headless agentic orchestration platform** that transforms the paradigm from interactive AI coding to autonomous background production service. It eliminates the human-in-the-loop dependency by translating standard project management artifacts (GitHub Issues, Epics, Kanban) into automated Execution Orders.

The system autonomously clones repositories, configures environments, generates and modifies code, runs test suites, and submits fully formatted Pull Requests—all without human intervention beyond initial task specification.

**Key Innovation:** The system is **Self-Bootstrapping**—it uses its own orchestration capabilities to build and refine itself, allowing the AI to "build its own house" while residing within it.

## Goals

- **Zero-Touch Construction**: User opens a specification issue and receives a functional, test-passed PR
- **Autonomous Development Floor**: Transform GitHub Issues into Execution Orders without manual prompting
- **Self-Bootstrapping Evolution**: System builds its own features using its orchestration workflows
- **Production-Grade Reliability**: Resilient polling, self-healing, and graceful degradation
- **Transparency & Auditability**: Complete visibility via GitHub Issues, labels, and comments

## Application Features

### Core Capabilities
- **Secure Webhook Ingestion** - HMAC SHA256 validation for GitHub events
- **Intelligent Event Triage** - Automatic template detection and labeling
- **Resilient Task Polling** - 60s polling with exponential backoff
- **Concurrency Control** - Assign-then-verify locking via GitHub Assignees
- **Shell-Bridge Execution** - Script-first DevContainer invocation
- **Autonomous Code Generation** - Full-stack implementation with test verification
- **Self-Healing** - Automatic recovery from crashes and stale states
- **Cost Guardrails** - Budget monitoring and stall protection (deferred)

### System Architecture

#### Four Pillars
1. **The Ear (Work Event Notifier)** - FastAPI webhook receiver with signature validation
2. **The State (Work Queue)** - GitHub Issues as distributed state (Markdown-as-a-Database)
3. **The Brain (Sentinel Orchestrator)** - Async polling service with lifecycle management
4. **The Hands (Opencode Worker)** - Isolated DevContainer with LLM agent execution

#### Key Components
- `src/notifier_service.py` - FastAPI webhook ingestion
- `src/orchestrator_sentinel.py` - Background polling and dispatch
- `src/models/work_item.py` - Unified data models and credential scrubbing
- `src/queue/github_queue.py` - ITaskQueue implementation
- `scripts/devcontainer-opencode.sh` - Shell bridge to worker environment
- `local_ai_instruction_modules/*.md` - Decoupled AI behavior logic

- [ ] Labels transition correctly (queued → in-progress → success/error)

- [ ] Non-zero exit from `devcontainer-opencode.sh up` detected
- [ ] Issue labeled as agent:infra-failure

### Implementation Failure Handling
- [ ] Non-zero exit from `devcontainer-opencode.sh prompt` detected
- [ ] Issue labeled as agent:impl-error
- [ ] Last 20 lines of stderr posted as sanitized comment

### Successful Execution
- [ ] Zero exit code detected from worker
- [ ] Changes committed and pushed to remote branch
- [ ] Pull Request created linking to original issue
- [ ] Issue labeled as agent:success

### Security & Payload Rejection
- [ ] Invalid X-Hub-Signature-256 returns 401
- [ ] Missing signature returns 401
- [ ] No JSON parsing before signature validation
- [ ] Only verified GitHub events processed

### Concurrency Control
- [ ] Multiple Sentinel instances don't claim same issue
- [ ] Race conditions handled gracefully
- [ ] Assign-then-verify pattern prevents collisions

### Heartbeat & Observability
- [ ] Heartbeat comments posted every 5 minutes during long tasks
- [ ] SENTINEL_ID included in all logs
- [ ] Worker output captured to JSONL files
- [ ] Credentials scrubbed from public comments

### Graceful Shutdown
- [ ] SIGTERM/SIGINT handled cleanly
- [ ] Current task allowed to finish
- [ ] No orphaned agent:in-progress issues on restart

## Risk Mitigation Strategies

| Risk | Impact | Mitigation |
|------|--------|------------|
| GitHub API Rate Limiting | High | Use GitHub App tokens (5K req/hr), implement local caching, use long-polling intervals, jittered exponential backoff |
| LLM Looping/Hallucination | High | Implement max_steps timeout, cost guardrails (deferred), add agent:retries counter, stall after >3 retries |
| Concurrency Collisions | Medium | Use GitHub Assignees as distributed lock, implement assign-then-verify pattern, graceful abort on race |
| Container Drift | Medium | Stop worker container between tasks, prevent state bleed, maintain clean environment |
| Security Injection | Medium | Strict HMAC signature validation, network isolation, ephemeral credentials, credential scrubbing |
| Network/Server Downtime | Medium | Polling-first architecture ensures self-healing, state reconciliation on restart |
| Resource Exhaustion | Medium | Strict CPU (2) and RAM (4GB) limits per worker, prevent DoS from rogue agents |

## Implementation Notes

### Key Assumptions
- GitHub App is configured with appropriate permissions (issues, PRs, contents, packages)
- Webhook secret (WEBHOOK_SECRET) is securely configured
- Bot account (SENTINEL_BOT_LOGIN) exists and is available for assignments
- Host server has sufficient resources (Docker, network access)
- ZHIPU_API_KEY has adequate quota for development and testing

### Critical Dependencies
- opencode CLI (v1.2.24+) must be available in devcontainer
- devcontainer-opencode.sh script must handle all Docker orchestration
- GitHub REST API must be accessible from host network
- uv package manager must be installed in base Docker image

### Architectural Decisions (ADRs)
- **ADR 07**: Shell-bridge execution over direct Docker SDK
- **ADR 08**: Polling-first with webhook optimization
- **ADR 09**: Provider-agnostic ITaskQueue interface

### Self-Bootstrapping Philosophy
Once Phase 1 is complete and the Sentinel service is running, the system will autonomously implement Phase 2 and Phase 3 by picking up its own task tickets. This demonstrates the core value proposition: the AI builds its own features using its orchestration capabilities.

### Deferred Features
- Cross-repo org-wide polling (single repo via GITHUB_REPO for now)
- Cost guardrails and budget monitoring
- Configurable environment reset modes (hardcoded to "stop")
- Advanced hierarchical task delegation refinements
